# Critique: 임시스케쥴 ROI 제안서 (056-temp-schedule-20-proposals-roi.md)

| Date | Handoff | Request | Summary |
|------|---------|---------|---------|

---

## Value Statement Assessment ✅

> "As a 사용자(특히 ADHD 사용자), I want 임시스케쥴을 '빠르게 잡고, 쉽게 수정하고, 필요하면 안전하게 정규 작업으로 전환'할 수 있어, so that 계획-실행 흐름이 끊기지 않고 작은 성공을 계속 쌓을 수 있다."

**평가**: Value Statement가 명확하고 ADHD 친화 목표와 직결됨. "So that" 절이 사용자 가치(흐름 유지, 작은 성공)를 구체적으로 표현.

---

## 5가지 강점 (Strengths)

| # | 강점 | 근거 |
|---|------|------|
| **S1** | **ROI 계산법이 단순하고 일관됨** | Impact(1-3)/Effort(S/M/L) 휴리스틱으로 20개 항목을 동일 기준으로 평가. 비교 가능. |
| **S2** | **ADHD 친화 가치가 항목별로 명시** | 각 항목에 "Expected value(ADHD)" 섹션이 있어 사용자 가치와 기능의 연결이 투명함. |
| **S3** | **Quick Wins 선별이 적절함** | ROI 3.0인 호버 퀵버튼, 승격 후 처리 옵션을 Quick Win으로 정확히 선별. |
| **S4** | **Scope 제약이 명확히 선언됨** | "Frontend-only" 제약이 문서 상단에 명시되어 백엔드 혼선 방지. |
| **S5** | **Risks/Tradeoffs 섹션이 존재** | 반복 이동 UX 복잡도, 렌더 비용, 단축키 충돌 등 5가지 리스크를 사전에 식별. |

## 8가지 구체적 개선안 (Improvements)

| # | 이슈 | 개선 제안 |
|---|------|----------|
| **I1** | **"아카이브" 정의 모호** | 항목1 "삭제/아카이브/그대로"에서 "아카이브"의 UI/데이터 처리가 불명확. → **아카이브 상태가 별도 필드/뷰인지, 단순 숨김인지 명시 필요.** |
| **I2** | **"초안(Draft) 모드" 구현 범위 과대** | 항목2의 드래그-생성 초안 모드는 저장 타이밍 변경, 취소 로직, UI 상태 분기가 필요해 Effort M 이상일 수 있음. → **Effort 재검토 또는 MVP 범위 축소(예: "저장 전 확인 모달"로 대체).** |
| **I3** | **"인라인 편집" 범위 미정의** | 항목3 "이름/시간/색"만 명시되어 있으나, 인라인 수정 UI가 리스트/타임라인 중 어디인지 불명확. → **적용 뷰(Day/Week/List) 명시.** |
| **I4** | **"충돌 하이라이트" 기준 미정의** | 항목4에서 "겹치면 색/패턴"이라고 했으나, 겹침 판정 기준(시간 범위 1분 이상? 같은 슬롯?)이 없음. → **충돌 조건 명시(예: 5분 이상 겹침 시 경고).** |
| **I5** | **"인박스로 보내기"와 기존 승격의 관계** | 항목5와 항목1(승격)이 별개로 나열되어 있으나, 승격 후 인박스 전송도 가능한지 흐름이 불명확. → **승격/인박스 전송/삭제의 관계를 플로우차트로 정리.** |
| **I6** | **반복 일정 선택 UX 미설계** | 항목6 "이번 1회만 vs 전체 패턴"은 Effort L로 평가되었으나, 선택 UI 형태(모달? 인라인? 툴팁?)가 없음. → **MVP는 "이번만 이동" 기본값 + 토스트 안내로 축소 가능.** |
| **I7** | **명령 팔레트 충돌 정책 누락** | 항목 3-1 "명령 팔레트"는 Risks에서 "단축키 충돌"만 언급. → **기존 앱 단축키(N/D/W/M/T)와 팔레트 트리거 키의 충돌 방지 정책 명시 필요.** |
| **I8** | **접근성 항목(3-4, 3-5) 검증 기준 없음** | 색각/포커스 강화 항목에 "WCAG 수준" 또는 테스트 시나리오가 없음. → **WCAG 2.1 AA 기준 명시 + axe-core 검증 단계 추가.** |

## 누락된 고ROI 아이디어 (3개)

| # | 아이디어 | 이유 |
|---|----------|------|
| **M1** | **임시블록 "복제 후 수정" 단축키** | 유사 일정을 연속 생성할 때 컨텍스트 메뉴 복제보다 `Ctrl+D`→즉시 편집이 더 빠름. Effort S / Impact Med. |
| **M2** | **"지금 빈 슬롯에 추가" 원클릭 버튼** | 현재 시간 기준 다음 빈 30분 슬롯에 즉시 블록 생성. 드래그 없이 가능. Effort S / Impact High. |
| **M3** | **임시블록 "일괄 삭제/정리" (범위 선택)** | 오래된 임시블록이 쌓이면 개별 삭제가 번거로움. 날짜 범위/완료 상태 기준 일괄 정리. Effort M / Impact Med. |

## 제약 위반 우려 항목 (Constraint Violations)

| 항목 | 위반 우려 | 권고 |
|------|----------|------|
| **항목 1-9 "포커스 시작 연결"** | `focusStore`와 연결은 가능하나, 타이머 시작 시 Electron IPC(`window focus`, 알림)가 개입할 수 있음. | **UI 버튼만 추가하고 타이머 호출은 기존 함수 재사용으로 한정. IPC 변경 없음 명시.** |
| **항목 2-3 "스마트 기본값"** | "다음 빈 슬롯" 계산이 메인 스케줄 데이터 접근을 요구할 수 있음(Dexie 조회는 허용). | **Dexie 조회만 사용, Firebase 실시간 조회 금지 명시.** |
| **항목 3-1 "명령 팔레트"** | 팔레트 자체는 UI이나, 전역 단축키 등록이 Electron `globalShortcut`을 건드릴 위험. | **렌더러 내 keydown 리스너만 사용, main process 변경 없음 명시.** |

## Top 7 우선순위 제안 (Rationale 포함)

| 순위 | 항목 | ROI | 선정 이유 |
|------|------|-----|----------|
| **1** | 호버 퀵버튼 노출 (2-2) | 3.0 | Effort S, 발견성 즉시 개선, 기존 컨텍스트 메뉴 로직 재사용 가능. |
| **2** | 승격 후 임시 항목 처리 옵션 (1-1) | 3.0 | Effort S, 승격 UX 마찰 해소, 기존 `promoteToRealTask` 확장으로 구현. |
| **3** | 충돌/겹침 하이라이트 (1-4) | 2.0 | Effort S, 시각적 피드백으로 재계획 부담 감소. 렌더링 로직만 추가. |
| **4** | 단축키 치트시트 (3-2) | 2.0 | Effort S, 학습 비용 감소, 기존 단축키 변경 없이 오버레이만 추가. |
| **5** | 템플릿 즐겨찾기/고정 (1-7) | 2.0 | Effort S, 템플릿 접근 비용 감소, `db.systemState` 필드 추가로 구현. |
| **6** | "내일/다음주 미루기" 버튼 (1-8) | 2.0 | Effort S, 날짜 +1/+7 연산만 필요, 결정 피로 감소 효과 큼. |
| **7** | 월간 뷰 정보 밀도 조절 (2-4) | 2.0 | Effort S, 인지 부하 감소, 기존 컴포넌트 조건부 렌더링으로 처리. |

**제외 사유**: Effort M/L 항목(초안 모드, 인라인 편집, 반복 선택 등)은 Quick Wins 이후 Phase 2로 이동 권장.

## Risk Assessment

| 리스크 | 영향 | 완화 방안 |
|--------|------|----------|

## Recommendations

1. **OPEN QUESTION 해결**: Target Release 확정 후 Quick Wins 3개 먼저 구현.
2. **항목별 상세 명세 보강**: I1~I8 개선안을 Planner가 반영하여 모호성 제거.
3. **누락 아이디어 검토**: M1~M3 중 "지금 빈 슬롯에 추가"(M2)는 ROI 높아 Quick Wins 후보로 재검토.
4. **제약 위반 방지**: 1-9, 2-3, 3-1 항목에 "IPC/백엔드 변경 없음" 명시 추가.
