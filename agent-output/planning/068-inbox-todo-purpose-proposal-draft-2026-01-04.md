## A) 1페이지 요약

### Value Statement and Business Objective
As a 사용자(특히 ADHD 사용자), I want 인박스/할일을 **키보드로 즉시 열고**, **짧은 triage 세션에서 3~5개만 빠르게 정리**하고, **오늘 일정으로 실패 없이 전환**할 수 있어, so that 계획/실행 흐름이 끊기지 않고 “머리 속 → 시스템” 전환이 즉시 이뤄진다.

### 현상(분석 기반 병목)
- **모달 전용 UX**: 접근이 TopToolbar CTA에 의존 → “즉시 캡처/즉시 정리” 동선이 길어짐
- **글로벌 오픈 핫키 부재**: 키보드 중심 플로우가 끊김(특히 집중 상태에서 마우스 이동 부담)
- **Triage ESC 미구현(카피/행동 불일치)**: UI 카피는 “ESC=종료”를 암시하나 실제로 triage 종료가 되지 않음
- **빠른 배치(오늘/내일/다음) 실패가 토스트 중심**: 슬롯 가용성/대안이 사전 노출되지 않아 ‘실패→다음 행동’이 불명확
- **ADHD 관점**: 세션형 미시 목표/시각적 단서가 약해 “짧게 끝내기”가 어렵고 피로 누적

### 핵심 우선순위 Top 5
1) **Undo/안전장치(삭제·이동·배치·변환 포함)**: 실수 공포를 제거해 triage 채택률을 올림 (P0)
2) **인박스 열기/닫기 렌더러 전역 핫키**: “즉시 캡처/즉시 정리” 진입 마찰 제거 (P0)
3) **Triage ESC 종료(실동작) + HUD/`?` 도움말**: 학습/탈출 비용을 낮춤 (P0)
4) **슬롯 추천 + 폴백 통합(사전 추천 + 실패 대안)**: 배치 실패를 “다음 행동”으로 연결 (P0)
5) **접근성 핵심(ARIA listbox/option + 고대비 포커스/선택 표시)**: 키보드/SR 사용자가 “기능을 쓸 수 있게” 만듦 (P0)

## A-1) KPI 측정/계측 원칙(관측 가능성 확보)

- KPI는 **UI 이벤트(eventBus) 카운터** 또는 **Dexie `db.systemState`에 집계값 기록** 방식으로 “관측 가능”해야 함.
- **localStorage는 금지(테마 키만 예외)**. 분석/계측을 위해서도 localStorage에 저장하지 않음.
- 세션 상태(예: triage 목표/선택)는 **기본은 메모리 스코프**(Zustand 등)로 유지하고, 영속화 확장은 별도 결정(ADR)로 분리.

## D) 리스크/트레이드오프 (5개)
1) **단축키 충돌/IME 이슈**: 입력 중 단축키 오작동 위험 → IME/입력 포커스 가드와 modalStack 우선순위 규칙 필요
2) **UI 복잡도 증가**: 기능을 추가할수록 ADHD 사용자는 더 혼란스러울 수 있음 → 기본은 단순, 고급은 펼치기(점진 공개) 필수
3) **모달 vs 패널 이원화**: 패널/모달 두 경로가 생기면 유지보수 부담↑ → 패널은 P2로 연기하고 읽기 전용 MVP부터 검증
4) **‘추천 슬롯’ 신뢰도**: 추천이 자주 실패하면 UX 역효과 → 실패 대안(내일/다음)과 원인 메시지(간단) 제공
5) **Undo 범위 확장 리스크**: 삭제뿐 아니라 이동/배치/변환까지 포함 시 상태 동기화/에지 케이스 증가 → “마지막 1회”로 범위를 제한하고 이벤트 로깅으로 관측
